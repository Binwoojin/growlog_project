# 📅 GrowLog 개발일지 — 2026.07.20

## 1. 오늘의 개발 목표

회원가입과 로그인 기능을 완료한 이후 GrowLog의 핵심 기능 중 하나인 **목표 관리 기능** 구현을 시작하였다.

목표 기능은 사용자가 자신의 성장 방향을 설정하고, 이후 작성할 성장 기록을 특정 목표와 연결하기 위한 기능이다.

오늘의 주요 개발 목표는 다음과 같다.

```text
GOAL 테이블 구조 확인
→ Member와 Goal의 연관관계 설정
→ Goal Entity 작성
→ GoalRepository 작성
→ GoalService 작성
→ GoalController 작성
→ 로그인 회원의 목표 등록
→ 로그인 회원의 목표 목록 조회
→ Goal JSP 화면 연결
```

목표 기능에서 필요한 기본 요구사항을 정리하였다.

### 목표 등록

* 목표 시작일과 종료일 설정

* 로그인한 회원의 목표만 조회

* 목표 목록 페이지에서 여러 목표 출력
* 목표 상태 및 기간 표시

### 추후 구현 예정

* 진행률 및 기록 수 표시

## 3. MEMBER와 GOAL의 관계 설정

한 명의 회원은 여러 개의 목표를 등록할 수 있으므로 다음과 같은 관계로 설계하였다.

```text
MEMBER 1 : N GOAL
```

관계의 의미는 다음과 같다.

```text
회원 한 명
→ 여러 개의 목표 등록 가능

목표 하나
→ 한 명의 회원에게 소속
```

목표 데이터가 어느 회원의 것인지 구분하기 위해 `GOAL` 테이블에 회원 번호를 외래키로 두었다.

```text
GOAL.member_no
→ MEMBER.member_no 참조
```

## 4. Goal Entity 작성

`GOAL` 테이블과 연결하기 위한 `Goal` Entity를 작성하였다.

목표 데이터에는 다음과 같은 정보가 포함되도록 구성하였다.

```text
goalNum
member
title
description
startDate
endDate
status
createdAt
updatedAt
```

회원과 목표의 다대일 관계를 표현하기 위해 `@ManyToOne`을 적용하였다.

```java
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "member_no", nullable = false)
private Member member;
```

이를 통해 여러 Goal 객체가 하나의 Member 객체를 참조하도록 구성하였다.

---

## 5. Member Entity의 연관관계 검토

Member Entity에서도 회원이 등록한 목표 목록을 관리할 수 있도록 양방향 관계 적용 여부를 검토하였다.

양방향 관계를 적용하는 경우 다음과 같이 구성할 수 있다.

```java
@OneToMany(
        mappedBy = "member",
        cascade = CascadeType.ALL,
        orphanRemoval = true
)
private List<Goal> goals = new ArrayList<>();
```

다만 현재 단계에서는 목표를 조회할 때 `GoalRepository`에서 회원 번호를 기준으로 조회할 수 있으므로, 반드시 Member Entity에 목표 목록을 추가하지 않아도 기능 구현이 가능하다는 점을 확인하였다.

필요한 방향의 관계만 우선 적용하고, 실제 사용 목적이 생겼을 때 양방향 관계를 추가하는 방향으로 검토하였다.

---

## 6. GoalRepository 작성

Goal Entity의 저장과 조회를 담당하는 Repository를 작성하였다.

```java
public interface GoalRepository
        extends JpaRepository<Goal, Long> {
}
```

기본적인 저장, 단건 조회, 전체 조회, 삭제 기능은 `JpaRepository`가 제공하는 메서드를 사용할 수 있다.

```text
save()
findById()
findAll()
delete()
deleteById()
```

---

## 7. 회원별 목표 조회 메서드 추가

목표 목록은 전체 목표가 아니라 현재 로그인한 회원의 목표만 조회해야 한다.

이를 위해 Member의 회원 번호를 기준으로 Goal을 조회하는 Repository 메서드를 작성하였다.

예시는 다음과 같다.

```java
List<Goal> findByMemberMemberNo(Long memberNo);
```

이 메서드명은 다음 객체 구조를 따라간다.

```text
Goal.member
→ Member.memberNo
```

즉, Goal Entity의 `member` 필드 안에 있는 `memberNo` 값을 기준으로 조회하는 파생 쿼리이다.

목표 정렬이 필요한 경우 다음과 같이 작성하는 방법도 검토하였다.

```java
List<Goal> findByMemberMemberNoOrderByCreatedAtDesc(
        Long memberNo
);
```

이를 사용하면 로그인 회원이 최근에 등록한 목표부터 조회할 수 있다.

## 8. Goal 요청 DTO 구성

목표 등록 화면의 입력값을 전달받기 위해 `GoalRequest` DTO를 구성하였다.

```java
public class GoalRequest {

    private String title;
    private String description;
    private LocalDate startDate;
    private LocalDate endDate;
}
```

목표 등록 요청에는 회원 번호를 직접 포함하지 않도록 하였다.

회원 번호를 화면에서 전달받으면 다른 회원의 번호로 요청을 조작할 가능성이 있으므로, 로그인 세션에서 현재 회원 정보를 가져와 목표와 연결하는 구조를 사용하였다.

```text
목표 내용
→ GoalRequest에서 전달

목표 소유 회원
→ HttpSession의 loginMember에서 확인
```

---

## 9. GoalService 작성

목표 등록과 회원별 조회를 담당하는 `GoalService`를 작성하였다.

### 목표 등록 처리

목표를 등록할 때는 다음 순서로 처리하도록 구성하였다.

```text
로그인 회원 확인
→ GoalRequest 입력값 확인
→ Goal Entity 생성
→ Member와 Goal 연결
→ Repository 저장
```

예시는 다음과 같다.

```java
public Goal createGoal(
        GoalRequest request,
        Member loginMember
) {
    Goal goal = Goal.builder()
            .member(loginMember)
            .title(request.getTitle())
            .description(request.getDescription())
            .startDate(request.getStartDate())
            .endDate(request.getEndDate())
            .build();

    return goalRepository.save(goal);
}
```

목표의 회원 정보는 요청 DTO가 아니라 세션에서 가져온 로그인 회원 객체를 사용하였다.

## 10. 회원별 목표 조회 Service 구현

로그인 회원의 회원 번호를 기준으로 목표 목록을 조회하는 Service 메서드를 작성하였다.

```java
public List<Goal> findGoalsByMember(
        Long memberNo
) {
    return goalRepository
            .findByMemberMemberNo(memberNo);
}
```

Controller는 Repository에 직접 접근하지 않고 Service를 통해 목표 목록을 조회하도록 구성하였다.

```text
GoalController
→ GoalService
→ GoalRepository
→ Database
```

## 11. GoalController 작성

목표 기능의 요청을 처리하기 위해 `GoalController`를 작성하였다.

공통 URL은 다음과 같이 설정하였다.

```java
@Controller
@RequiredArgsConstructor
@RequestMapping("/goal")
public class GoalController {

    private final GoalService goalService;
}
```

목표 관련 페이지는 `/goal`을 기준으로 관리하도록 구성하였다.

```text
/goal
→ 목표 목록

/goal/write
→ 목표 등록 화면

/goal/create
→ 목표 등록 처리
```

실제 URL은 Controller 구성에 따라 이후 조정할 수 있도록 하였다.

## 12. 세션에서 로그인 회원 가져오기

Goal 기능은 회원 전용 기능이므로 Controller에서 세션에 저장된 로그인 회원 정보를 가져오도록 하였다.

```java
private Member getLoginMember(
        HttpSession session
) {
    return (Member) session.getAttribute(
            "loginMember"
    );
}
```

각 메서드에서 반복해서 형변환하지 않도록 별도의 메서드로 분리하였다.

```java
Member loginMember =
        getLoginMember(session);
```

## 13. 목표 목록 페이지 접근 제어

로그인하지 않은 사용자가 `/goal` 주소에 직접 접근할 수 있으므로 목표 목록을 조회하기 전에 로그인 세션을 확인하였다.

```java
@GetMapping
public String goalList(
        HttpSession session,
        Model model
) {
    Member loginMember =
            getLoginMember(session);

    if (loginMember == null) {
        return "redirect:/login";
    }

    List<Goal> goalList =
            goalService.findGoalsByMember(
                    loginMember.getMemberNo()
            );

    model.addAttribute(
            "goalList",
            goalList
    );

    return "goal/list";
}
```

세션에 로그인 회원이 없으면 로그인 페이지로 이동하도록 처리하였다.

---

## 14. 로그인 회원 기준 목표 조회

목표 목록 조회 시 전체 Goal 데이터를 가져오는 방식은 사용하지 않았다.

```java
goalRepository.findAll();
```

전체 조회를 사용하면 다른 회원이 등록한 목표까지 함께 노출될 가능성이 있기 때문이다.

대신 세션의 로그인 회원 번호를 Service로 전달하였다.

```java
goalService.findGoalsByMember(
        loginMember.getMemberNo()
);
```

이를 통해 현재 로그인한 회원의 목표만 조회하도록 구현하였다.

## 15. 목표 등록 페이지 연결

사용자가 목표를 입력할 수 있도록 목표 등록 화면으로 이동하는 GET 요청을 구성하였다.

```java
@GetMapping("/write")
public String goalWrite(
        HttpSession session
) {
    Member loginMember =
            getLoginMember(session);

    if (loginMember == null) {
        return "redirect:/login";
    }

    return "goal/write";
}
```

목표 등록 화면 역시 회원 전용 기능이므로 로그인 세션을 검증하도록 하였다.

## 16. 목표 등록 처리 구현

목표 등록 화면에서 전달된 내용을 받아 DB에 저장하는 POST 요청을 구성하였다.

```java
@PostMapping
public String createGoal(
        @ModelAttribute GoalRequest request,
        HttpSession session
) {
    Member loginMember =
            getLoginMember(session);

    if (loginMember == null) {
        return "redirect:/login";
    }

    goalService.createGoal(
            request,
            loginMember
    );

    return "redirect:/goal";
}
```

목표 저장 후에는 목표 목록 페이지로 리다이렉트하여 새로 등록한 목표를 확인할 수 있도록 하였다.

## 17. 목표 등록 JSP 구성

목표 등록 화면인 `goal/write.jsp`를 구성하였다.

입력 항목은 다음과 같다.

```text
목표 제목
목표 설명
시작일
종료일
카테고리
```

기본적인 form 구조는 다음과 같이 구성하였다.

```jsp
<form
    action="${pageContext.request.contextPath}/goal"
    method="post"
>
    <input
        type="text"
        name="title"
        placeholder="목표 제목"
    >

    <textarea
        name="description"
        placeholder="목표 설명"
    ></textarea>

    <input
        type="date"
        name="startDate"
    >

    <input
        type="date"
        name="endDate"
    >

    <button type="submit">
        목표 등록
    </button>
</form>
```

입력 필드의 `name` 속성은 `GoalRequest`의 필드명과 일치하도록 구성하였다.

## 18. 목표 목록 JSP 구성

로그인 회원의 목표 목록을 출력하기 위해 `goal/list.jsp`를 구성하였다.

Controller에서 전달한 목표 목록을 JSTL 반복문으로 출력하도록 하였다.

```jsp
<c:forEach
    var="goal"
    items="${goalList}"
>
    <div class="goal-card">
        <h3>${goal.title}</h3>

        <p>${goal.description}</p>

        <span>
            ${goal.startDate}
            ~
            ${goal.endDate}
        </span>
    </div>
</c:forEach>
```

목표가 없는 경우 별도의 안내 메시지를 표시하는 구조도 검토하였다.

```jsp
<c:if test="${empty goalList}">
    <p>아직 등록된 목표가 없습니다.</p>
</c:if>
```

## 20. 목표 기능의 사용자 데이터 보호

Goal 기능 구현 과정에서 회원 번호를 URL이나 form의 hidden 값으로 전달하는 것보다 로그인 세션을 기준으로 처리하는 것이 안전하다고 판단하였다.

다음과 같은 방식은 사용하지 않았다.

```jsp
<input
    type="hidden"
    name="memberNo"
    value="${loginMember.memberNo}"
>
```

클라이언트에서 전달하는 회원 번호는 수정될 수 있기 때문이다.

대신 서버의 세션에서 로그인 회원을 확인하였다.

```java
Member loginMember =
        (Member) session.getAttribute(
                "loginMember"
        );
```

이를 통해 목표의 소유자를 서버에서 결정하도록 구성하였다.

## 21. 목표 날짜 처리 검토

목표에는 시작일과 종료일이 포함되므로 날짜 데이터 타입과 검증 방법을 검토하였다.

Java에서는 날짜만 필요한 경우 `LocalDate`를 사용하였다.

```java
private LocalDate startDate;
private LocalDate endDate;
```

JSP에서는 `input type="date"`를 사용해 `yyyy-MM-dd` 형식으로 값을 전달하도록 하였다.

```jsp
<input type="date" name="startDate">
<input type="date" name="endDate">
```

추후 다음과 같은 날짜 검증을 추가하기로 하였다.

```text
시작일이 종료일보다 늦지 않은지 확인
시작일과 종료일이 비어 있지 않은지 확인
목표 상태와 날짜의 관계 확인
```

## 22. 목표 상태 관리 방향 검토

목표가 현재 어떤 상태인지 구분하기 위한 상태값을 검토하였다.

예정한 상태는 다음과 같다.

```text
PLANNED
→ 시작 전

IN_PROGRESS
→ 진행 중

COMPLETED
→ 완료

PAUSED
→ 잠시 중단
```

초기 목표 등록 시 기본 상태를 `IN_PROGRESS` 또는 시작일에 따라 결정하는 방식을 검토하였다.

현재 단계에서는 기본적인 목표 등록과 조회 기능을 먼저 구현하고, 상태 변경 기능은 수정 기능과 함께 추가하기로 하였다.

## 23. 오늘 발생한 트러블슈팅

목표 기능을 구현하면서 다음 사항을 확인하고 수정하였다.

### Goal 페이지 세션 검증

로그인 여부를 확인하지 않으면 비로그인 사용자도 `/goal`에 접근할 수 있으므로 Controller에서 세션을 우선 확인하도록 수정하였다.

### 회원별 목표 조회

전체 Goal 데이터를 조회하면 다른 회원의 목표가 노출될 수 있으므로 로그인 회원 번호를 기준으로 조회하도록 수정하였다.

상세한 내용은 별도의 `Troubleshooting - 2026.07.20` 문서에 정리하였다.

## 26. 오늘 배운 점

* 회원별 데이터를 다룰 때는 전체 조회가 아니라 로그인 회원을 기준으로 조회해야 한다.
* 클라이언트가 전달한 회원 번호보다 서버 세션의 로그인 정보를 신뢰하는 것이 안전하다.
* JPA 파생 쿼리는 연관 객체의 필드까지 따라가 조회 조건을 만들 수 있다.
* 회원 전용 Controller에서는 비즈니스 로직을 실행하기 전에 로그인 세션을 먼저 확인해야 한다.
* Entity의 연관관계는 실제 조회 방향과 사용 목적을 고려하여 설정해야 한다.
* 화면의 입력 필드명과 DTO의 필드명이 일치해야 데이터 바인딩이 정상적으로 처리된다.
* 목표 날짜는 문자열보다 `LocalDate`로 관리하는 것이 날짜 비교와 검증에 유리하다.
* 기능을 한 번에 완성하기보다 등록과 조회부터 구현한 후 수정과 삭제를 추가하는 방식이 개발 과정을 관리하기 쉽다.
