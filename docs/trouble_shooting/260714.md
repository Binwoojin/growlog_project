# 📅 Troubleshooting - 2026.07.14

## 1. Repository 메서드명 오류로 인한 `UnsatisfiedDependencyException`

### 📌 문제

Spring Boot 애플리케이션을 실행하는 과정에서 다음과 같은 오류가 발생하며 서버가 정상적으로 시작되지 않았다.

```text
UnsatisfiedDependencyException
```

Controller가 의존하는 Service를 생성하지 못했고, Service가 사용하는 Repository의 Bean도 정상적으로 등록되지 않았다.

---

### 🔍 원인

MemberRepository에 작성한 이메일 중복 확인 메서드의 이름이 Spring Data JPA의 메서드 명명 규칙과 일치하지 않았다.

```java
boolean existByEmail(String email);
```

Spring Data JPA에서는 데이터 존재 여부를 확인하는 파생 쿼리 메서드를 작성할 때 `existBy`가 아니라 `existsBy`를 사용해야 한다.

잘못된 메서드명으로 인해 Spring이 `exist`라는 속성을 기준으로 쿼리를 생성하려고 했고, Member Entity에서 해당 속성을 찾지 못해 Repository Bean 생성에 실패했다.

Repository Bean 생성 실패는 Service와 Controller의 의존성 주입 실패로 이어져 최종적으로 `UnsatisfiedDependencyException`이 발생하였다.

---

### 🛠 해결 방법

Repository 메서드명을 Spring Data JPA의 규칙에 맞게 수정하였다.

```java
public interface MemberRepository
        extends JpaRepository<Member, Long> {

    boolean existsByEmail(String email);
}
```

Service에서 호출하는 메서드명도 동일하게 변경하였다.

```java
public boolean isEmailDuplicated(String email) {
    return memberRepository.existsByEmail(email);
}
```

---

### ✅ 결과

* MemberRepository Bean 정상 생성
* MemberService와 MemberController 의존성 주입 성공
* Spring Boot 애플리케이션 정상 실행
* 이메일 중복 확인 기능 정상 동작

---

### 💡 배운 점

* Spring Data JPA의 파생 쿼리는 정해진 메서드 명명 규칙을 따라야 한다.
* `UnsatisfiedDependencyException`이 발생했을 때 가장 바깥쪽 Controller만 확인하지 않고, 오류 메시지의 `Caused by`를 따라 Repository까지 확인해야 한다.
* 의존성 주입 오류는 실제로는 Repository 메서드명이나 Entity 필드명 문제에서 시작될 수 있다.

## 2. 회원가입 페이지 접속 시 404 오류

### 📌 문제

브라우저에서 회원가입 페이지로 이동하려고 했지만 다음과 같은 404 오류가 발생하였다.

```text
Whitelabel Error Page
Type: Not Found
Status: 404
```

회원가입 버튼이나 `/join` 주소를 통해 페이지에 접근해도 `join.jsp` 화면이 표시되지 않았다.

---

### 🔍 원인

회원가입 페이지를 반환하는 GET 요청 매핑이 Controller에 없거나, 실제 접근 주소와 Controller에 작성한 경로가 일치하지 않았다.

회원가입 처리를 위한 POST 메서드만 구현되어 있더라도 회원가입 화면을 처음 열기 위한 GET 매핑이 없으면 `/join` 요청을 처리할 수 없다.

또한 Controller가 반환하는 View 이름과 실제 JSP 파일 경로가 일치하는지도 함께 확인해야 했다.

---

### 🛠 해결 방법

MemberController에 회원가입 화면을 반환하는 GET 매핑을 추가하였다.

```java
@Controller
@RequiredArgsConstructor
public class MemberController {

    @GetMapping("/join")
    public String joinPage() {
        return "join";
    }
}
```

JSP 파일이 View Resolver가 찾을 수 있는 위치에 존재하는지도 확인하였다.

```text
src
└─ main
   └─ webapp
      └─ WEB-INF
         └─ views
            └─ join.jsp
```

Controller의 반환값은 파일 전체 경로가 아닌 View 이름만 사용하였다.

```java
return "join";
```

View Resolver 설정에 따라 실제로는 다음 경로와 결합된다.

```text
/WEB-INF/views/ + join + .jsp
```

---

### ✅ 결과

* `/join` 주소의 GET 요청이 Controller와 정상 연결됨
* 회원가입 페이지 정상 출력
* 회원가입 화면과 회원가입 처리 요청의 역할 분리

---

### 💡 배운 점

* 페이지를 보여주는 GET 요청과 데이터를 처리하는 POST 요청은 별도로 매핑해야 한다.
* 404 오류가 발생하면 URL, Controller 매핑, 반환 View 이름, JSP 실제 위치를 순서대로 확인해야 한다.
* Spring MVC에서는 Controller가 반환하는 문자열이 JSP 파일명이 아니라 View 이름이라는 점을 이해해야 한다.

## 3. 로그아웃 요청 시 405 오류

### 📌 문제

로그아웃 기능을 실행했을 때 다음과 같은 오류가 발생하였다.

```text
Method Not Allowed
Status: 405
```

로그아웃 URL은 존재했지만 브라우저에서 전달한 HTTP 요청 방식과 Controller가 허용한 요청 방식이 일치하지 않았다.

---

### 🔍 원인

로그아웃 링크는 일반적으로 `<a>` 태그로 구현되어 있었기 때문에 GET 요청을 전송하였다.

```html
<a href="/logout">로그아웃</a>
```

하지만 Controller에서는 로그아웃을 POST 요청으로만 처리하도록 작성되어 있었다.

```java
@PostMapping("/logout")
public String logout(HttpSession session) {
    session.invalidate();
    return "redirect:/login";
}
```

따라서 `/logout`이라는 URL 자체는 존재하지만 GET 요청을 처리하는 메서드가 없어 405 오류가 발생하였다.

404는 요청 주소를 처리할 매핑 자체가 없는 경우이고, 405는 주소는 존재하지만 현재 요청 방식이 허용되지 않는 경우라는 차이가 있었다.

---

### 🛠 해결 방법

현재 화면에서 `<a>` 태그를 사용하고 있었기 때문에 Controller의 로그아웃 매핑을 GET 방식으로 맞추었다.

```java
@GetMapping("/logout")
public String logout(HttpSession session) {
    session.invalidate();
    return "redirect:/login";
}
```

또는 POST 방식을 유지하려면 링크 대신 form을 사용할 수 있다.

```html
<form action="${pageContext.request.contextPath}/logout"
      method="post">
    <button type="submit">로그아웃</button>
</form>
```

당시 프로젝트에서는 구현 구조에 맞춰 요청 방식과 Controller 매핑을 일치시키는 방향으로 수정하였다.

---

### ✅ 결과

* 로그아웃 요청 정상 처리
* 기존 로그인 세션 무효화
* 로그아웃 후 로그인 페이지로 정상 이동
* GET과 POST 요청 방식 불일치 문제 해결

---

### 💡 배운 점

* 405 오류가 발생하면 URL뿐 아니라 HTTP 요청 방식도 함께 확인해야 한다.
* `<a>` 태그는 기본적으로 GET 요청을 전송한다.
* `@GetMapping`과 `@PostMapping`은 화면에서 전송하는 요청 방식과 정확히 일치해야 한다.
* 로그아웃처럼 세션 상태를 변경하는 기능은 이후 보안을 강화할 때 POST 방식으로 전환하는 것도 고려할 수 있다.

---

## 📝 2026.07.14 트러블슈팅 요약

이날은 회원가입과 로그인 기능의 기본 구조를 구현하면서 Spring Data JPA의 메서드 명명 규칙, Spring MVC의 URL 매핑, HTTP 요청 방식에 관한 오류를 해결하였다.

MemberRepository의 `existByEmail`을 `existsByEmail`로 수정하여 Repository Bean 생성 오류를 해결하였고, 회원가입 페이지를 표시하는 GET 매핑을 추가하여 404 오류를 해결하였다.

또한 로그아웃 화면에서 전송하는 요청 방식과 Controller의 매핑 방식이 달라 발생한 405 오류를 수정하였다.

이를 통해 Spring 오류를 해결할 때 다음 순서로 확인하는 방법을 익혔다.

```text
오류 메시지와 Caused by 확인
→ Controller 요청 경로 확인
→ HTTP 요청 방식 확인
→ Service 호출 확인
→ Repository 메서드명 확인
→ Entity 필드명 확인
```
