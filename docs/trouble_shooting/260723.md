# 📅 Troubleshooting — 2026.07.23

## 1. Home의 최근 성장 기록이 의도한 디자인과 다르게 출력되는 문제

### 📌 문제

Home Dashboard에 최근 성장 기록 데이터를 연결한 뒤, 데이터 자체는 정상적으로 출력되었지만 카드의 배치와 정보 구조가 처음 설계한 화면과 다르게 나타났다.

특히 다음과 같은 문제가 있었다.

```text
기록 카드 내부 요소의 정렬 불일치
제목과 부가 정보의 간격 부족
기록 내용이 카드 안에서 어색하게 배치됨
기존 Home 디자인과 최근 기록 영역의 통일감 부족
```

원하는 화면은 기록 제목, 작성일, 기록 종류, 내용 요약이 명확하게 구분된 카드 형태였지만 실제 출력 화면에서는 정보가 한곳에 몰려 보였다.

---

### 🔍 원인

백엔드 데이터를 화면에 출력하는 것에 우선순위를 두면서, 기존 HTML 구조에 JSTL 반복문과 EL 표현식만 추가하였다.

이 과정에서 데이터는 정상적으로 출력되었지만 다음 요소가 충분히 구분되지 않았다.

```text
기록 카드 전체 영역
기록 유형 및 날짜 영역
제목 영역
내용 미리보기 영역
상세 페이지 이동 영역
```

또한 기존 정적 화면을 실제 반복 데이터 구조로 변경하면서 카드 하나를 기준으로 작성한 CSS가 여러 개의 데이터가 반복되는 환경과 정확히 맞지 않았다.

---

### 🛠 해결 방법

최근 성장 기록 하나를 독립된 카드로 구분하고 카드 내부 구조를 다시 정리하였다.

```jsp
<c:forEach
    items="${recentRecords}"
    var="recordItem"
>
    <a
        href="${pageContext.request.contextPath}/record/${recordItem.recordNum}"
        class="recent-record-card"
    >
        <div class="recent-record-card__top">
            <!-- 기록 유형 및 날짜 -->
        </div>

        <h3 class="recent-record-card__title">
            ${recordItem.recordTitle}
        </h3>

        <p class="recent-record-card__content">
            ${recordItem.recordContent}
        </p>
    </a>
</c:forEach>
```

CSS에서도 카드와 카드 내부 요소의 역할을 분리하였다.

```css
.recent-record-card {
    display: block;
    padding: 20px;
    border: 1px solid #dde6dc;
    border-radius: 18px;

    color: inherit;
    text-decoration: none;
}

.recent-record-card__top {
    display: flex;
    align-items: center;
    justify-content: space-between;
}

.recent-record-card__content {
    overflow: hidden;

    display: -webkit-box;
    -webkit-box-orient: vertical;
    -webkit-line-clamp: 2;
}
```

---

### ✅ 결과

* 최근 성장 기록이 카드별로 명확하게 구분됨
* 제목, 날짜, 기록 종류, 내용의 시각적 우선순위 개선
* 긴 기록 내용이 카드 영역을 벗어나지 않음
* Home Dashboard 전체 디자인과 자연스럽게 통일됨
* 카드를 선택하면 성장 기록 상세 페이지로 이동 가능

---

### 💡 배운 점

* 데이터가 정상적으로 출력되는 것과 화면이 의도대로 구성되는 것은 별개의 문제이다.
* 정적 HTML을 반복 데이터 구조로 변경할 때는 카드 한 개가 아니라 여러 데이터가 반복되는 상황을 기준으로 CSS를 작성해야 한다.
* 카드 내부의 정보는 의미별로 영역을 분리해야 이후 디자인 수정과 유지보수가 쉬워진다.
* 긴 본문은 Dashboard에서 모두 보여주기보다 줄 수를 제한하여 미리보기 형태로 제공하는 것이 적절하다.

## 2. Controller의 Model 이름과 JSP 참조 이름 불일치 점검

### 📌 문제

Home Dashboard에서 최근 목표나 최근 성장 기록 데이터를 조회했음에도 JSP에서 값이 출력되지 않거나 빈 영역처럼 보일 가능성이 있었다.

데이터 조회 결과는 정상이어도 다음 연결 과정 중 하나가 일치하지 않으면 화면에는 아무것도 출력되지 않는다.

```text
Service 반환값
→ Controller 지역 변수
→ Model Attribute 이름
→ JSP의 items 이름
→ 반복문 내부 변수 이름
```

---

### 🔍 원인

Controller에서 데이터를 저장한 Model 이름과 JSP에서 사용하는 이름이 다르거나, `<c:forEach>`에서 선언한 변수와 내부 EL 표현식의 이름이 일치하지 않으면 데이터가 출력되지 않는다.

예를 들어 Controller에서 다음과 같이 저장했을 때,

```java
model.addAttribute("recentGoals", recentGoals);
```

JSP에서 다른 이름을 사용하면 값이 조회되지 않는다.

```jsp
<c:forEach items="${goalList}" var="goalItem">
```

반복 변수도 마찬가지이다.

```jsp
<c:forEach
    items="${recentGoals}"
    var="goalItem"
>
    ${goal.goalTitle}
</c:forEach>
```

위 코드에서는 반복 변수로 `goalItem`을 선언했지만 내부에서는 `goal`을 참조하고 있다.

---

### 🛠 해결 방법

Controller와 JSP에서 사용하는 이름을 일관되게 통일하였다.

#### Controller

```java
model.addAttribute(
        "recentGoals",
        recentGoals
);

model.addAttribute(
        "recentRecords",
        recentRecords
);
```

#### 최근 목표 JSP

```jsp
<c:forEach
    items="${recentGoals}"
    var="goalItem"
>
    ${goalItem.goalTitle}
</c:forEach>
```

#### 최근 성장 기록 JSP

```jsp
<c:forEach
    items="${recentRecords}"
    var="recordItem"
>
    ${recordItem.recordTitle}
</c:forEach>
```

값이 없을 때 단순히 빈 화면으로 보이지 않도록 빈 목록 처리도 추가하였다.

```jsp
<c:if test="${empty recentRecords}">
    <p>아직 작성한 성장 기록이 없습니다.</p>
</c:if>
```

---

### ✅ 결과

* Controller에서 전달한 최근 목표 데이터 정상 출력
* 최근 성장 기록 데이터 정상 출력
* 반복문 내부에서 잘못된 객체명을 참조하는 문제 방지
* 데이터가 없는 경우에도 안내 화면 표시
* Home Dashboard의 데이터 출력 구조 안정화

---

### 💡 배운 점

* JSP 데이터 출력 오류는 Repository만 확인할 것이 아니라 전체 전달 과정을 순서대로 점검해야 한다.
* `model.addAttribute()`의 이름은 JSP에서 사용하는 EL 이름과 정확히 일치해야 한다.
* `<c:forEach>`의 `var`는 반복문 내부에서 사용할 객체의 이름이다.
* 데이터가 없어서 출력되지 않는 경우와 이름이 잘못되어 출력되지 않는 경우를 구분해야 한다.

점검 순서는 다음과 같이 정리할 수 있다.

```text
Repository 조회 결과
→ Service 반환 결과
→ Controller Model 이름
→ JSP items 이름
→ forEach var 이름
→ Entity 실제 필드명
```

## 3. Home의 연속 출석 카드가 클릭되지 않는 문제

### 📌 문제

Home Dashboard의 연속 출석 카드가 `<article>` 태그로 구성되어 있어 카드 디자인은 정상적으로 표시되었지만, 카드를 클릭해도 Attendance 페이지로 이동하지 않았다.

기존 구조는 다음과 같았다.

```jsp
<article class="summary-card">
    <!-- 연속 출석 정보 -->
</article>
```

`article` 태그는 독립적인 콘텐츠 영역을 나타낼 뿐 페이지 이동 기능은 제공하지 않는다.

---

### 🔍 원인

Home의 다른 메뉴처럼 연속 출석 카드도 클릭할 수 있을 것으로 예상했지만, 카드에 `<a>` 태그나 JavaScript 이동 이벤트가 연결되어 있지 않았다.

즉, 시각적으로는 버튼이나 링크처럼 보였지만 실제 HTML 구조에는 이동 기능이 없었다.

---

### 🛠 해결 방법

카드 전체를 `<a>` 태그로 감싸 Attendance 페이지로 이동하도록 수정하였다.

```jsp
<a
    href="${pageContext.request.contextPath}/mypage/attendance"
    class="summary-card-link"
>
    <article class="summary-card">
        <!-- 연속 출석 정보 -->
    </article>
</a>
```

링크를 적용하면서 기존 카드 디자인이 바뀌지 않도록 기본 링크 스타일을 제거하였다.

```css
.summary-card-link {
    display: block;

    color: inherit;
    text-decoration: none;
}
```

마우스를 올렸을 때 클릭할 수 있는 카드라는 점이 드러나도록 효과도 적용할 수 있다.

```css
.summary-card-link:hover .summary-card {
    transform: translateY(-2px);
}
```

---

### ✅ 결과

* 연속 출석 카드 전체 영역 클릭 가능
* `/mypage/attendance` 페이지 이동 구조 완성
* 기존 카드 디자인 유지
* Home에서 Attendance로 이어지는 사용자 흐름 개선

---

### 💡 배운 점

* `<article>`은 의미를 표현하는 시맨틱 태그이며 링크 기능은 없다.
* 카드 전체를 클릭하도록 만들 때는 `<a>` 태그로 감싸는 방식이 가장 명확하다.
* `<a>` 태그를 적용하면 기본 글자 색상과 밑줄이 생길 수 있으므로 CSS 초기화가 필요하다.
* 링크 경로는 실제 Controller 매핑과 반드시 일치해야 한다.

## 4. Attendance를 독립 페이지로 구현할지 MyPage 하위 페이지로 구성할지 혼선

### 📌 문제

Home의 연속 출석 카드를 Attendance 페이지와 연결하려고 했지만 Attendance를 다음 중 어느 구조로 만들지 명확하지 않았다.

```text
/attendance
```

또는

```text
/mypage/attendance
```

Attendance를 독립 메뉴로 구현하면 이동 경로는 단순하지만, 출석 기록은 개인 활동 정보이므로 MyPage의 계정 설정 및 개인정보 메뉴와 화면 구조가 분리될 수 있었다.

---

### 🔍 원인

기능 구현 순서만 고려했을 때는 Attendance 페이지를 먼저 독립적으로 만드는 것이 쉬워 보였다.

하지만 전체 서비스 구조를 다시 검토한 결과 Attendance에는 다음과 같은 개인 데이터가 포함된다.

* 현재 연속 기록
* 최고 연속 기록
* 월별 기록 날짜
* 이번 달 출석 횟수
* 향후 획득한 뱃지

따라서 Attendance는 공용 기능보다 개인 활동 내역에 가까웠고, MyPage와의 관계를 먼저 정리할 필요가 있었다.

---

### 🛠 해결 방법

Attendance를 MyPage의 하위 메뉴로 구성하기로 결정하였다.

```text
/mypage
→ 마이페이지 메인

/mypage/profile
→ 개인정보 수정

/mypage/account
→ 계정 설정

/mypage/attendance
→ 출석 기록
```

Controller도 `/mypage`를 공통 경로로 사용하는 구조를 검토하였다.

```java
@Controller
@RequestMapping("/mypage")
public class MyPageController {

    @GetMapping
    public String myPage() {
        return "mypage/index";
    }

    @GetMapping("/attendance")
    public String attendancePage() {
        return "mypage/attendance";
    }
}
```

Home의 연속 출석 카드도 최종 URL 구조에 맞춰 연결하였다.

```jsp
href="${pageContext.request.contextPath}/mypage/attendance"
```

---

### ✅ 결과

* MyPage와 Attendance의 관계 명확화
* URL 구조 일관성 확보
* Attendance를 개인 활동 관리 기능으로 분류
* 추후 프로필, 계정 설정, 출석 기록을 하나의 공통 레이아웃에서 제공 가능
* Home 카드 링크의 최종 경로 결정

---

### 💡 배운 점

* 기능을 먼저 구현하기 전에 서비스 내에서 해당 기능이 어느 메뉴에 속하는지 정리해야 한다.
* URL 구조는 단순한 주소가 아니라 서비스의 기능 계층을 표현한다.
* 개인 데이터와 관련된 기능은 MyPage 하위에서 관리하면 사용자 흐름이 자연스럽다.
* 공통 레이아웃이 필요한 페이지는 Controller와 JSP 폴더 구조도 같은 계층으로 구성하는 것이 좋다.

## 5. MyPage 요구사항 확장으로 Controller 구조를 다시 설계한 문제

### 📌 문제

처음에는 Attendance 페이지를 연결하기 위한 최소한의 MyPage Controller만 고려하였다.

하지만 MyPage에는 출석 기록 외에도 다음 기능이 필요하다는 점을 확인하였다.

```text
프로필 정보
개인정보 수정
계정 설정
비밀번호 변경
출석 기록
```

Attendance 연결만을 기준으로 Controller를 작성하면 추후 다른 메뉴를 추가할 때 URL과 메서드 구조를 다시 수정해야 할 가능성이 있었다.

---

### 🔍 원인

MyPage를 단일 화면으로만 생각하고 Controller를 단순하게 구성하려 했기 때문이다.

하지만 실제 MyPage는 여러 개인 설정 기능을 포함하는 상위 영역이며, Attendance는 그중 하나의 하위 메뉴이다.

---

### 🛠 해결 방법

MyPage를 상위 기능으로 정의하고 하위 페이지를 확장할 수 있도록 Controller 구조를 다시 설계하였다.

```java
@Controller
@RequiredArgsConstructor
@RequestMapping("/mypage")
public class MyPageController {

    @GetMapping
    public String myPage(
            HttpSession session,
            Model model
    ) {
        Member loginMember =
                getLoginMember(session);

        if (loginMember == null) {
            return "redirect:/login";
        }

        model.addAttribute(
                "loginMember",
                loginMember
        );

        return "mypage/index";
    }

    @GetMapping("/attendance")
    public String attendance(
            HttpSession session,
            Model model
    ) {
        Member loginMember =
                getLoginMember(session);

        if (loginMember == null) {
            return "redirect:/login";
        }

        return "mypage/attendance";
    }

    private Member getLoginMember(
            HttpSession session
    ) {
        return (Member) session.getAttribute(
                "loginMember"
        );
    }
}
```

추후 다음 요청을 같은 Controller 또는 역할별 Controller로 확장할 수 있도록 구성하였다.

```text
GET  /mypage/profile
POST /mypage/profile

GET  /mypage/password
POST /mypage/password

GET  /mypage/attendance
```

---

### ✅ 결과

* MyPage를 상위 개인 관리 영역으로 정의
* Attendance를 하위 메뉴로 자연스럽게 연결
* 개인정보 및 계정 설정 기능을 추가할 수 있는 구조 확보
* MyPage 관련 URL의 일관성 향상
* 로그인 세션 검증 구조 재사용 가능

---

### 💡 배운 점

* Controller는 현재 필요한 메서드만 보는 것이 아니라 기능의 확장 가능성도 고려하여 설계해야 한다.
* 상위 메뉴와 하위 메뉴가 있는 경우 공통 `@RequestMapping`을 활용하면 URL을 일관되게 관리할 수 있다.
* 회원 전용 페이지는 모든 요청에서 로그인 세션을 검증해야 한다.
* 동일한 세션 조회 코드가 반복되면 공통 메서드나 인터셉터로 분리할 수 있다.

최근 성장 기록 영역은 데이터가 출력되는 것에 그치지 않고 의도한 카드 디자인에 맞도록 HTML과 CSS 구조를 다시 구성하였다.

Home 데이터 출력 과정에서는 Controller의 Model 이름, JSP의 `items`, 반복 변수의 `var`, 내부 EL 표현식이 모두 일치하는지 점검하였다.

연속 출석 카드는 `<article>` 태그만 사용해 이동할 수 없었던 문제를 `<a>` 태그로 감싸는 방식으로 해결하였다.

또한 Attendance 페이지를 독립 기능으로 구성할지 고민했지만, 개인 활동 데이터라는 특성을 고려해 MyPage의 하위 메뉴로 구성하기로 결정하였다. 이에 따라 MyPage Controller 역시 개인정보, 계정 설정, 출석 기록을 확장할 수 있는 구조로 다시 설계하였다.

오늘의 핵심 학습 내용은 다음과 같다.

```text
반복 데이터에 맞는 Dashboard 카드 구조 설계
Controller와 JSP 사이의 데이터 이름 통일
시맨틱 태그와 링크 태그의 역할 구분
서비스 계층에 맞는 URL 구조 설계
확장 가능한 MyPage Controller 구성
```
