# 📅 Troubleshooting — 2026.07.22

## 1. 성장 기록 상세 페이지의 본문이 들여쓰기되어 출력되는 문제

### 📌 문제

성장 기록 상세 페이지에서 본문을 출력했을 때 첫 문장이 왼쪽에서 바로 시작하지 않고, 앞에 많은 공백이 들어간 것처럼 표시되었다.

사용자가 작성 화면에서는 정상적으로 입력했지만 상세 페이지에서는 다음과 같이 본문의 시작 부분이 과도하게 들여쓰기 된 형태로 출력되었다.

```text
                오늘 목표를 완료했다.
다음 목표도 꾸준히 진행할 예정이다.
```

---

### 🔍 원인

JSP에서 본문 데이터를 출력할 때 EL 표현식을 태그 내부에서 여러 줄로 작성하고 있었다.

```jsp
<div class="record-content">
    ${record.content}
</div>
```

CSS에서 다음과 같이 공백과 줄바꿈을 유지하도록 설정한 상태였다.

```css
white-space: pre-wrap;
```

`white-space: pre-wrap`은 사용자가 입력한 줄바꿈뿐 아니라 HTML 문서에 작성된 공백과 줄바꿈도 유지한다.

따라서 JSP 코드에서 `${record.content}` 앞에 작성된 개행과 들여쓰기 공백이 실제 화면에도 반영되면서 본문이 들여쓰기 된 것처럼 표시되었다.

---

### 🛠 해결 방법

본문을 출력하는 EL 표현식 앞뒤에 불필요한 공백이나 줄바꿈이 들어가지 않도록 태그를 한 줄로 작성하였다.

```jsp
<div class="record-content">${record.content}</div>
```

사용자가 작성한 본문의 줄바꿈은 유지하면서 긴 문장이 영역 밖으로 넘어가지 않도록 CSS를 적용하였다.

```css
.record-content {
    white-space: pre-wrap;
    word-break: break-word;
}
```

---

### ✅ 결과

* 본문 첫 줄 앞의 불필요한 공백 제거
* 사용자가 작성한 줄바꿈 유지
* 긴 문장이 화면 영역에 맞게 자동으로 줄바꿈됨
* 상세 페이지의 본문 가독성 개선

---

### 💡 배운 점

* `white-space: pre-wrap`은 사용자 입력뿐 아니라 HTML 코드의 공백과 줄바꿈도 화면에 반영할 수 있다.
* JSP에서 여러 줄의 텍스트를 출력할 때는 EL 표현식 주변의 공백도 확인해야 한다.
* 데이터 자체에 공백이 있는지, JSP 출력 구조 때문에 공백이 생기는지를 구분해서 확인해야 한다.
* 화면 출력 오류가 발생하면 DB 데이터, JSP 구조, CSS 속성을 순서대로 확인하는 것이 좋다.

## 2. 성장 기록 목록 페이지 이동 시 `/record/list` 404 오류

### 📌 문제

성장 기록 수정 기능을 구현한 뒤 목록 페이지로 이동하려고 했지만 다음과 같은 404 오류가 발생하였다.

```text
There was an unexpected error
type=Not Found
status=404

No static resource record/list.
```

브라우저에서 `/record/list` 주소로 요청했지만 해당 요청을 처리하는 Controller를 찾지 못했다.

---

### 🔍 원인

Controller에서는 성장 기록 목록 페이지를 다음과 같이 매핑하고 있었다.

```java
@Controller
@RequestMapping("/record")
public class GrowthRecordController {

    @GetMapping
    public String recordList() {
        return "record/list";
    }
}
```

위 코드에서 실제 브라우저 요청 주소는 다음과 같다.

```text
/record
```

반면 `record/list`는 브라우저에서 요청할 URL이 아니라 View Resolver가 찾을 JSP의 이름이다.

```text
Controller 요청 URL
/record

반환 View 이름
record/list

실제 JSP
/WEB-INF/views/record/list.jsp
```

하지만 수정 및 상세 페이지의 링크 또는 리다이렉트에서 View 이름을 URL처럼 사용하였다.

```java
return "redirect:/record/list";
```

또는 다음과 같이 링크가 작성되어 있었다.

```jsp
<a href="${pageContext.request.contextPath}/record/list">
    목록으로
</a>
```

`/record/list`를 처리하는 `@GetMapping("/list")`이 존재하지 않았기 때문에 Spring은 Controller 매핑을 찾지 못했고, 이후 정적 리소스를 탐색하다가 `No static resource record/list` 오류를 반환하였다.

---

### 🛠 해결 방법

성장 기록 목록의 실제 Controller URL에 맞게 링크와 리다이렉트 주소를 `/record`로 통일하였다.

#### 수정 및 삭제 후 목록 이동

```java
return "redirect:/record";
```

#### JSP의 목록 버튼

```jsp
<a href="${pageContext.request.contextPath}/record">
    목록으로
</a>
```

Controller의 View 반환값은 기존대로 유지하였다.

```java
return "record/list";
```

---

### ✅ 결과

* 성장 기록 목록 페이지 정상 접근
* 수정 및 삭제 후 목록 페이지 정상 이동
* `No static resource record/list` 오류 해결
* Controller URL과 JSP View 이름의 역할 구분

---

### 💡 배운 점

Spring MVC에서 다음 세 가지 경로는 서로 다른 의미를 가진다.

```text
/record
→ 브라우저가 요청하는 URL

record/list
→ Controller가 반환하는 View 이름

/WEB-INF/views/record/list.jsp
→ 실제 JSP 파일 경로
```

* `return "record/list"`와 `redirect:/record/list`는 서로 완전히 다른 동작이다.
* 404 오류에 `No static resource`가 표시되더라도 정적 파일 문제라고 단정하지 말고 Controller 매핑을 먼저 확인해야 한다.
* 페이지 이동 경로는 JSP 파일 위치가 아니라 Controller의 요청 매핑을 기준으로 작성해야 한다.

## 3. 성장 기록 수정·삭제 시 다른 회원의 기록에 접근할 수 있는 문제 방지

### 📌 문제

성장 기록 수정과 삭제 기능은 URL을 통해 기록 번호를 전달받는다.

```text
/record/1/edit
/record/1/delete
```

기록 번호만으로 데이터를 조회할 경우 사용자가 주소의 번호를 임의로 변경하여 다른 회원의 기록에 접근할 가능성이 있었다.

예를 들어 로그인한 회원의 기록 번호가 `5`더라도 다음 주소를 직접 입력할 수 있다.

```text
/record/1/edit
```

단순히 `findById(recordNum)`만 사용하면 기록의 작성자를 확인하지 않고 데이터가 조회될 수 있다.

---

### 🔍 원인

초기 수정 및 삭제 구조에서는 기록 번호를 기준으로만 Entity를 조회하는 방식을 고려하였다.

```java
growthRecordRepository.findById(recordNum);
```

그러나 GrowLog는 여러 회원이 사용하는 서비스이므로 기록 번호가 존재한다는 것만으로 수정이나 삭제 권한이 있다고 판단할 수 없다.

다음 두 조건을 함께 확인해야 했다.

```text
해당 기록이 존재하는가?
현재 로그인한 회원이 작성한 기록인가?
```

---

### 🛠 해결 방법

수정과 삭제 요청을 처리할 때 기록 번호와 세션의 로그인 회원 번호를 함께 Service로 전달하였다.

```java
growthRecordService.updateRecord(
        recordNum,
        loginMember.getMemberNo(),
        request
);
```

```java
growthRecordService.deleteRecord(
        recordNum,
        loginMember.getMemberNo()
);
```

Service에서는 조회한 기록의 작성자와 로그인 회원을 비교하였다.

```java
GrowthRecord record =
        growthRecordRepository.findById(recordNum)
                .orElseThrow(() ->
                        new IllegalArgumentException(
                                "성장 기록을 찾을 수 없습니다."
                        )
                );

if (!record.getMember()
        .getMemberNo()
        .equals(memberNo)) {

    throw new IllegalArgumentException(
            "해당 기록에 대한 권한이 없습니다."
    );
}
```

검증에 성공한 경우에만 수정 또는 삭제를 실행하였다.

```java
growthRecordRepository.delete(record);
```

상세 조회와 수정 페이지 접근에도 같은 소유권 검증 방식을 적용하였다.

---

### ✅ 결과

* 로그인한 회원은 자신의 성장 기록만 상세 조회 가능
* 자신의 성장 기록만 수정 가능
* 자신의 성장 기록만 삭제 가능
* URL의 기록 번호를 변경해도 다른 회원의 데이터에 접근할 수 없음
* 성장 기록 CRUD의 데이터 보호 강화

---

### 💡 배운 점

* URL에 포함된 PK 값만으로 사용자의 권한을 판단해서는 안 된다.
* 데이터 조회 성공과 해당 데이터에 대한 작업 권한은 별도의 문제이다.
* 회원별 데이터를 수정하거나 삭제할 때는 로그인 회원과 데이터 소유자를 반드시 비교해야 한다.
* 상세 조회에도 소유권 검증이 필요하며, 수정과 삭제에서만 검사해서는 부족하다.
* 동일한 소유권 검증이 반복될 경우 Repository 조회 조건이나 공통 검증 메서드로 분리할 수 있다.

예를 들어 Repository 단계에서 기록 번호와 회원 번호를 함께 조회하는 방식도 사용할 수 있다.

```java
Optional<GrowthRecord>
findByRecordNumAndMemberMemberNo(
        Long recordNum,
        Long memberNo
);
```

이 방식을 사용하면 조회 단계에서부터 현재 회원의 기록만 가져올 수 있다.

## 📝 2026.07.22 트러블슈팅 요약

오늘은 성장 기록 상세 조회, 수정, 삭제 기능을 구현하면서 화면 출력, URL 매핑, 데이터 접근 권한과 관련된 문제를 해결하였다.

본문 출력 과정에서는 `white-space: pre-wrap`과 JSP 코드의 공백이 함께 적용되어 첫 줄이 들여쓰기 된 것처럼 표시되는 문제를 수정하였다.

성장 기록 목록 이동 과정에서는 JSP의 View 이름인 `record/list`를 브라우저 요청 URL로 사용하여 발생한 404 오류를 확인하였다. 실제 Controller 매핑인 `/record`로 링크와 리다이렉트 경로를 통일하여 해결하였다.

또한 성장 기록 번호만으로 수정과 삭제를 처리하지 않고, 세션의 로그인 회원 번호와 기록 소유자를 함께 확인하도록 수정하여 다른 회원의 데이터에 접근할 수 없도록 하였다.

