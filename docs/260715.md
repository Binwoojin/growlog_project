회원 기능 완료 후에는 GrowLog의 성장 기록에 이미지와 유튜브 영상을 첨부하기 위한 `MEDIA` 테이블 구조를 다시 검토하였다.

## 2. Gmail SMTP 설정 수정

회원가입 시 이메일 인증번호를 전송하기 위해 Gmail SMTP 연결 설정을 점검하였다.

기존 설정에서는 SMTP 서버 주소가 잘못 작성되어 메일 서버에 연결하지 못하는 문제가 있었다.

잘못된 설정은 다음과 같았다.

```yaml
host: stmp.gmail.com
```

이를 Gmail의 올바른 SMTP 주소로 수정하였다.

```yaml
spring:
  mail:
    host: smtp.gmail.com
    port: 587
    username: Gmail 주소
    password: Google 앱 비밀번호
    properties:
      mail:
        smtp:
          auth: true
          starttls:
            enable: true
```

Google 계정의 일반 비밀번호가 아닌 앱 비밀번호를 사용하도록 설정하고 메일 전송 기능을 다시 테스트하였다.

## 3. 이메일 인증번호 발송 기능 확인

회원가입 화면에서 이메일을 입력한 뒤 인증번호 발송 버튼을 누르면 서버에서 6자리 인증번호를 생성하도록 구성하였다.

```text
100000 ~ 999999
```

생성한 인증번호는 사용자가 입력한 이메일 주소로 발송하고, 검증에 필요한 정보는 세션에 저장하였다.

```text
EMAIL_KEY
→ 인증을 요청한 이메일

CODE_KEY
→ 서버에서 생성한 인증번호

EXPIRES_AT
→ 인증번호 만료 시각

VERIFIED
→ 이메일 인증 완료 여부
```

인증번호의 유효시간은 5분으로 설정하였다.

## 4. 이메일 인증번호 확인 기능 연결

사용자가 회원가입 화면에서 인증번호를 입력하면 세션에 저장된 인증번호와 비교하도록 구현하였다.

인증번호 확인 과정에서는 다음 항목을 검사하였다.

```text
인증번호 발송 이력 존재 여부
→ 입력한 인증번호와 발송한 인증번호 일치 여부
→ 인증번호 유효시간 만료 여부
→ 발송한 이메일과 현재 입력한 이메일 일치 여부
```

인증에 성공하면 세션의 인증 완료 상태를 변경하였다.

```text
VERIFIED = true
```

회원가입 처리 시에도 이메일 인증 완료 여부를 다시 확인하도록 구성하여, 화면에서만 인증된 것처럼 처리되는 것을 방지하였다.

## 5. 회원가입 데이터 전달 구조 점검

회원가입 과정에서 사용자의 이름을 입력했지만 `MEMBER` 테이블의 `USER_NAME` 컬럼에 값이 전달되지 않는 문제가 발생하였다.

다음 계층의 필드 연결 상태를 확인하였다.

```text
join.jsp
→ JoinRequest DTO
→ MemberService
→ Member Entity
→ MEMBER 테이블
```

각 계층에서 사용하는 필드명을 `userName`으로 통일하였다.

### 회원가입 입력 필드

```jsp
<input type="text" name="userName">
```

### JoinRequest DTO

```java
public class JoinRequest {

    private String email;
    private String password;
    private String passwordConfirm;
    private String nickname;
    private String userName;
}
```

### Member Entity 생성

```java
Member member = Member.builder()
        .email(request.getEmail())
        .password(encodedPassword)
        .nickname(request.getNickname())
        .userName(request.getUserName())
        .build();
```

필드 연결을 수정한 후 사용자의 이름이 DB에 정상적으로 저장되는 것을 확인하였다.

## 8. 로그인 기능 테스트

회원가입을 통해 저장된 계정으로 로그인 기능을 테스트하였다.

로그인 처리 과정은 다음과 같이 구성하였다.

```text
이메일로 회원 조회
→ 회원 존재 여부 확인
```

```java
);
```

로그인에 성공한 회원 정보는 다음 세션 키로 저장하였다.

```java
session.setAttribute("loginMember", member);
```

## 9. Home 페이지 이동 및 세션 확인

로그인 성공 후 `/home`으로 정상 이동하는지 확인하였다.

HomeController에서는 세션에 로그인 회원 정보가 있는지 검사하였다.

```java
@GetMapping("/home")
public String homePage(HttpSession session) {

    Member loginMember =
            (Member) session.getAttribute("loginMember");

    if (loginMember == null) {
        return "redirect:/login";
    }

    return "home";
}
```

다음 상황을 각각 테스트하였다.

### 로그인한 상태

```text
/home 접근
→ Home 페이지 정상 출력
```

### 로그인하지 않은 상태

```text
/home 직접 접근
→ 로그인 페이지로 리다이렉트
```

이를 통해 세션 기반 접근 제어가 정상적으로 동작하는 것을 확인하였다.

## 12. 다음 기능 개발을 위한 ERD 검토

회원 기능을 완료한 뒤 성장 기록에 첨부할 이미지와 영상 데이터를 관리하기 위해 기존 ERD의 `MEDIA` 테이블을 검토하였다.

초기 `MEDIA` 테이블은 다음과 같이 구성되어 있었다.

```text
media_num
record_num
media_type
media_url
thumbnail_url
created_at
```

`media_type`은 다음 두 종류를 사용할 예정이었다.

```text
IMAGE
VIDEO
```

하지만 실제 서비스 요구사항을 검토하면서 다음 조건을 추가로 고려해야 했다.

* 하나의 성장 기록에 이미지 여러 개 첨부 가능
* 하나의 성장 기록에 이미지와 영상을 함께 첨부 가능
* 영상 파일을 직접 업로드하지 않음
* 영상은 유튜브 링크만 등록 가능
* 유튜브 영상의 썸네일 주소 저장 가능
* 미디어 순서 관리 필요 가능성

## 13. MEDIA 테이블 관계 재검토

성장 기록과 미디어의 관계를 다음과 같이 정의하였다.

```text
GROWTH_RECORD 1 : N MEDIA
```

하나의 성장 기록에는 여러 개의 미디어가 연결될 수 있으며, 각 미디어 행이 이미지 또는 유튜브 영상 하나를 나타내도록 설계하였다.

따라서 이미지와 영상이 동시에 등록되는 경우에도 하나의 행에 두 종류를 저장하는 것이 아니라 각각 별도의 행으로 저장하는 구조를 사용하기로 하였다.

예시는 다음과 같다.

```text
record_num = 1, media_type = IMAGE
record_num = 1, media_type = IMAGE
record_num = 1, media_type = VIDEO
```

이 구조를 통해 한 성장 기록에서 이미지와 유튜브 영상을 함께 관리할 수 있다.

## 14. MEDIA 테이블 개선안 정리

미디어 구조를 다음과 같은 방향으로 정리하였다.

```sql
CREATE TABLE media (
    media_num BIGINT AUTO_INCREMENT PRIMARY KEY,
    record_num BIGINT NOT NULL,
    media_type VARCHAR(20) NOT NULL,
    media_url VARCHAR(500) NOT NULL,
    thumbnail_url VARCHAR(500),
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,

    CONSTRAINT fk_media_record
        FOREIGN KEY (record_num)
        REFERENCES growth_record(record_num)
        ON DELETE CASCADE
);
```

### 컬럼 역할

| 컬럼              | 역할                |
| --------------- | ----------------- |
| `media_num`     | 미디어 식별 번호         |
| `record_num`    | 연결된 성장 기록 번호      |
| `media_type`    | 이미지 또는 영상 구분      |
| `media_url`     | 이미지 경로 또는 유튜브 URL |
| `thumbnail_url` | 영상 썸네일 URL        |
| `created_at`    | 미디어 등록 일시         |

영상은 직접 업로드하지 않고 유튜브 주소만 저장하기로 하였다.

이미지의 경우 `thumbnail_url`이 필요하지 않으므로 null을 허용하도록 구성하였다.

## 15. 미디어 저장 방식 결정

미디어 유형에 따라 `media_url`의 의미를 다르게 사용하기로 하였다.

### 이미지

```text
media_type = IMAGE
media_url = 업로드된 이미지의 저장 경로
thumbnail_url = NULL
```

### 유튜브 영상

```text
media_type = VIDEO
media_url = 유튜브 영상 주소
thumbnail_url = 유튜브 썸네일 주소
```

추후 백엔드에서는 `media_type`에 따라 입력값을 다르게 검증할 예정이다.

```text
IMAGE
→ 이미지 파일 및 이미지 경로 검증

VIDEO
→ 유튜브 주소 형식 검증
```

## 17. 오늘 발생한 트러블슈팅

개발 중 다음 문제를 확인하고 해결하였다.

### Gmail SMTP 연결 실패

SMTP 서버 주소의 오타와 Gmail 앱 비밀번호 설정을 수정하여 인증 메일이 정상 발송되도록 처리하였다.

### `USER_NAME` null 저장 오류

JSP, DTO, Service, Entity 사이의 필드 연결을 확인하고 `userName`으로 통일하여 회원 이름이 정상적으로 저장되도록 수정하였다.

### 회원 기능 전체 흐름 점검

이메일 인증부터 회원가입, 로그인, 세션 저장, Home 페이지 이동까지 단계별로 테스트하여 각 기능이 정상적으로 연결되었는지 확인하였다.

상세한 해결 과정은 별도의 `Troubleshooting - 2026.07.15` 문서에 정리하였다.

## 19. 다음 개발 계획

회원 기능이 완료되었으므로 다음 개발에서는 GrowLog의 핵심 기능인 목표 기능부터 구현할 예정이다.

```text
GOAL 테이블 생성
→ Goal Entity
→ GoalRepository
→ GoalService
→ GoalController
→ 목표 등록 화면
→ 회원별 목표 조회
→ 목표 수정 및 삭제
```

목표 기능 이후에는 성장 기록과 미디어 기능을 순서대로 구현할 계획이다.

## 20. 오늘 배운 점

* 외부 서비스 연결 오류는 코드뿐 아니라 서버 주소, 포트, 계정 보안 설정도 함께 확인해야 한다.
* 설정 파일의 작은 오타 하나가 기능 전체의 실패로 이어질 수 있다.
* HTML 입력값이 DB까지 전달되는 과정에서는 JSP, DTO, Service, Entity의 필드 연결을 모두 확인해야 한다.
* 회원 기능은 각각 따로 테스트하는 것뿐 아니라 전체 사용자 흐름을 기준으로 검증해야 한다.
* 비밀번호는 반드시 암호화하여 저장하고 로그인 시 `matches()`로 비교해야 한다.
* 세션 기반 로그인은 로그인 성공, 세션 저장, 페이지 접근 제어, 로그아웃까지 하나의 흐름으로 구현해야 한다.
* 하나의 성장 기록에 이미지와 영상이 함께 등록되더라도 미디어별로 행을 분리하면 유연하게 관리할 수 있다.
* 테이블 설계는 현재 기능뿐 아니라 이후 추가될 요구사항까지 고려하여 검토해야 한다.
