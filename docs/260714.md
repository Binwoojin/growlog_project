# 📅 GrowLog 개발일지 — 2026.07.14

## 1. 오늘의 개발 목표

GrowLog 서비스를 사용하기 위한 기본 회원 기능을 구현하였다.

이날의 주요 목표는 다음과 같다.

```text
회원가입 화면 연결
→ 회원가입 데이터 전달
→ 이메일·닉네임 중복 확인
→ 비밀번호 암호화
→ 로그인 처리
→ 세션 저장
→ 로그아웃 처리
→ 이메일 인증 기능 구성
```

회원가입과 로그인 기능은 이후 목표, 성장 기록, 타임라인 등의 데이터를 회원별로 구분하기 위한 기반이기 때문에 다른 기능보다 먼저 구현하였다.

---

## 2. 회원가입 화면 연결

회원가입 페이지인 `join.jsp`에 사용자가 입력해야 할 항목을 구성하였다.

### 회원가입 입력 항목

* 이메일
* 이메일 인증번호
* 비밀번호
* 비밀번호 확인
* 이름
* 닉네임

회원가입 화면에서 입력한 값이 Controller로 전달될 수 있도록 각 입력 요소의 `name` 속성을 DTO의 필드명과 연결하였다.

```jsp
<input type="email" name="email">

<input type="password" name="password">

<input type="password" name="passwordConfirm">

<input type="text" name="userName">

<input type="text" name="nickname">
```

회원가입 화면을 처음 표시하기 위한 GET 요청과 회원가입 데이터를 처리하는 POST 요청을 분리하였다.

```java
@GetMapping("/join")
public String joinPage() {
    return "join";
}
```

```java
@PostMapping("/join")
public String join(
        @ModelAttribute JoinRequest request) {

    memberService.join(request);

    return "redirect:/login";
}
```

## 3. 회원가입 요청 DTO 구성

회원가입 화면의 입력값을 전달받기 위해 `JoinRequest` DTO를 구성하였다.

```java
public class JoinRequest {

    private String email;
    private String password;
    private String passwordConfirm;
    private String userName;
    private String nickname;
}
```

Entity를 화면 요청에 직접 사용하지 않고 회원가입에 필요한 입력값만 DTO로 분리하였다.

이를 통해 다음과 같은 역할을 구분하였다.

```text
JoinRequest
→ 회원가입 화면에서 전달된 입력값 보관

Member
→ MEMBER 테이블과 연결되는 데이터 객체
```

## 4. 회원가입 Service 로직 구현

Controller에서 전달받은 `JoinRequest`를 이용하여 회원가입 처리 로직을 작성하였다.

회원가입 과정에서 다음 항목을 확인하도록 구성하였다.

* 이메일 중복 여부
* 닉네임 중복 여부
* 비밀번호와 비밀번호 확인 값의 일치 여부
* 비밀번호 암호화
* Member Entity 생성
* DB 저장

회원가입 요청을 Member Entity로 변환하였다.

```java
Member member = Member.builder()
        .email(request.getEmail())
        .password(encodedPassword)
        .nickname(request.getNickname())
        .userName(request.getUserName())
        .build();
```

완성된 Member 객체는 Repository를 통해 저장하였다.

```java
memberRepository.save(member);
```

## 5. 이메일 및 닉네임 중복 확인 구현

동일한 이메일이나 닉네임으로 여러 계정이 생성되지 않도록 중복 확인 기능을 구성하였다.

`MemberRepository`에 이메일과 닉네임 존재 여부를 확인하는 메서드를 추가하였다.

```java
public interface MemberRepository
        extends JpaRepository<Member, Long> {

    boolean existsByEmail(String email);

    boolean existsByNickname(String nickname);
}
```

Service에서는 Repository의 조회 결과를 이용하여 중복 여부를 판단하도록 하였다.

```java
public boolean isEmailDuplicated(String email) {
    return memberRepository.existsByEmail(email);
}
```

```java
public boolean isNicknameDuplicated(String nickname) {
    return memberRepository.existsByNickname(nickname);
}
```

회원가입 화면에서는 JavaScript와 중복 확인 API를 연결하여 사용자가 입력한 값이 이미 사용 중인지 확인할 수 있도록 구성하였다.

## 6. 비밀번호 암호화 적용

회원 비밀번호를 DB에 평문으로 저장하지 않도록 `PasswordEncoder`를 적용하였다.

회원가입 시 사용자가 입력한 비밀번호를 암호화한 뒤 Entity에 저장하도록 구현하였다.

```java
String encodedPassword =
        passwordEncoder.encode(request.getPassword());
```

```java
Member member = Member.builder()
        .email(request.getEmail())
        .password(encodedPassword)
        .nickname(request.getNickname())
        .userName(request.getUserName())
        .build();
```

DB에는 사용자가 입력한 비밀번호가 그대로 저장되지 않고 암호화된 문자열이 저장되도록 하였다.

## 7. 로그인 요청 DTO 구성

로그인 화면에서 이메일과 비밀번호를 전달받기 위해 `LoginRequest` DTO를 구성하였다.

```java
public class LoginRequest {

    private String email;
    private String password;
}
```

로그인 시 이메일을 기준으로 회원을 조회하고, 입력한 비밀번호와 DB에 저장된 암호화 비밀번호를 비교하도록 하였다.

```java
Member member =
        memberRepository.findByEmail(request.getEmail())
                .orElseThrow(() ->
                        new IllegalArgumentException(
                                "존재하지 않는 회원입니다."
                        ));
```

비밀번호 비교에는 `PasswordEncoder.matches()`를 사용하였다.

```java
boolean passwordMatches =
        passwordEncoder.matches(
                request.getPassword(),
                member.getPassword()
        );
```

## 8. 로그인 Controller 구현

로그인 페이지를 표시하는 GET 요청과 로그인 데이터를 처리하는 POST 요청을 구성하였다.

```java
@GetMapping("/login")
public String loginPage() {
    return "login";
}
```

로그인에 성공하면 회원 정보를 세션에 저장하고 Home 페이지로 이동하도록 구현하였다.

```java
@PostMapping("/login")
public String login(
        @ModelAttribute LoginRequest request,
        HttpSession session) {

    Member loginMember =
            memberService.login(request);

    session.setAttribute(
            "loginMember",
            loginMember
    );

    return "redirect:/home";
}
```

로그인 세션 키는 프로젝트 전체에서 다음 이름으로 통일하였다.

```text
loginMember
```

## 9. 로그아웃 기능 구현

로그인 상태를 종료하기 위해 로그아웃 기능을 추가하였다.

로그아웃 요청이 들어오면 현재 세션을 무효화하도록 처리하였다.

```java
@GetMapping("/logout")
public String logout(HttpSession session) {

    session.invalidate();

    return "redirect:/login";
}
```

로그아웃 이후에는 기존 세션에 저장된 회원 정보를 사용할 수 없으며 로그인 페이지로 이동하도록 구성하였다.

로그아웃 화면 또는 메뉴도 로그인 상태에서 접근할 수 있도록 연결하였다.

## 10. Home 페이지 접근 제어 확인

Home 페이지는 로그인한 회원만 접근할 수 있도록 세션 검증 로직을 적용하였다.

```java
@GetMapping("/home")
public String homePage(HttpSession session) {

    Member loginMember =
            (Member) session.getAttribute(
                    "loginMember"
            );

    if (loginMember == null) {
        return "redirect:/login";
    }

    return "home";
}
```

이를 통해 로그인하지 않은 사용자가 브라우저 주소창에 `/home`을 직접 입력하더라도 로그인 페이지로 이동하도록 하였다.

## 11. 이메일 인증 기능 구성 시작

회원가입 과정에서 사용자가 실제로 소유한 이메일인지 확인하기 위해 Gmail SMTP 기반 이메일 인증 기능을 구성하기 시작하였다.

이메일 인증 흐름은 다음과 같이 설계하였다.

```text
이메일 입력
→ 인증번호 발송 요청
→ 6자리 인증번호 생성
→ Gmail SMTP를 이용한 메일 전송
→ 인증번호를 세션에 저장
→ 사용자가 인증번호 입력
→ 세션 값과 비교
→ 인증 완료 처리
```

### 인증번호 생성

6자리 숫자 인증번호를 생성하도록 구성하였다.

```text
100000 ~ 999999
```

### 세션 저장 항목

이메일 인증 과정에서 다음 정보를 세션에 저장하도록 설계하였다.

```text
EMAIL_KEY
CODE_KEY
EXPIRES_AT
VERIFIED
```

각 항목의 역할은 다음과 같다.

| 세션 값         | 역할         |
| ------------ | ---------- |
| `EMAIL_KEY`  | 인증 요청한 이메일 |
| `CODE_KEY`   | 발송한 인증번호   |
| `EXPIRES_AT` | 인증번호 만료 시간 |
| `VERIFIED`   | 인증 성공 여부   |

인증번호의 유효시간은 5분으로 설정하였다.

## 12. Gmail SMTP 설정 추가

Spring Boot에서 이메일을 발송하기 위해 `application.yaml`에 Gmail SMTP 관련 설정을 추가하였다.

```yaml
spring:
  mail:
    host: smtp.gmail.com
    port: 587
    username: Gmail 주소
    password: Google 앱 비밀번호
    properties:
      mail:
        smtp:
          auth: true
          starttls:
            enable: true
```

Gmail SMTP를 사용하려면 Google 계정에서 다음 설정이 필요하다는 점을 확인하였다.

```text
Google 계정 2단계 인증 활성화
→ 앱 비밀번호 발급
→ application.yaml에 앱 비밀번호 등록
```

이날은 이메일 발송을 위한 구조와 설정을 구성하였으며, 실제 메일 전송 오류 수정과 최종 동작 확인은 다음 개발 과정에서 이어서 진행하였다.

## 16. 오늘 발생한 트러블슈팅

회원 기능을 구현하면서 다음 문제를 확인하고 해결하였다.

### Repository 메서드명 오류

이메일 존재 여부를 확인하는 메서드를 다음과 같이 잘못 작성하였다.

```java
existByEmail
```

Spring Data JPA의 메서드 명명 규칙에 맞게 수정하였다.

```java
existsByEmail
```

### 회원가입 페이지 404 오류

회원가입 페이지를 표시하기 위한 GET 매핑이 누락되어 `/join` 접속 시 404 오류가 발생하였다.

```java
@GetMapping("/join")
public String joinPage() {
    return "join";
}
```

GET 매핑을 추가하여 문제를 해결하였다.

### 로그아웃 405 오류

화면에서는 `<a>` 태그를 통해 GET 요청을 전송했지만 Controller는 POST 요청만 허용하고 있어 405 오류가 발생하였다.

화면 요청 방식과 Controller의 매핑 방식을 일치하도록 수정하였다.

상세한 해결 과정은 별도의 `Troubleshooting - 2026.07.14` 문서에 정리하였다.

## 19. 오늘 배운 점

* Spring MVC에서는 화면을 반환하는 GET 요청과 데이터를 처리하는 POST 요청을 구분해야 한다.
* Spring Data JPA의 파생 쿼리 메서드는 정해진 명명 규칙을 정확히 따라야 한다.
* 회원 비밀번호는 반드시 암호화하여 저장해야 한다.
* 로그인 기능은 회원 조회와 비밀번호 확인뿐 아니라 세션 저장까지 포함하여 구현해야 한다.
* `<a>` 태그와 `<form>`은 전송하는 HTTP 요청 방식이 다르므로 Controller 매핑과 일치시켜야 한다.
* JavaScript 입력 검증만으로는 부족하며 서버에서도 데이터를 다시 검증해야 한다.
* 이메일 인증 기능은 인증번호뿐 아니라 이메일, 만료시간, 인증 완료 상태를 함께 관리해야 한다.
* 외부 서비스와 연결되는 설정 정보는 오타와 보안 설정을 꼼꼼하게 확인해야 한다.
