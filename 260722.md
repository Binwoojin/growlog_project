# 📅 GrowLog 개발일지 — 2026.07.22

## 1. 오늘의 개발 목표

오늘은 기존에 구현한 성장 기록 등록 및 목록 조회 기능을 기반으로 다음 기능을 완성하는 것을 목표로 하였다.

```text
성장 기록 상세 조회
→ 상세 페이지 화면 구성
→ 성장 기록 수정
→ 성장 기록 삭제
→ 전체 CRUD 동작 테스트
```

기존 성장 기록 기능에는 등록과 조회만 구현되어 있었기 때문에, 수정과 삭제까지 추가하여 완전한 CRUD 구조를 만드는 것을 오늘의 목표로 설정하였다.

## 2. 성장 기록 상세 조회 기능 구현

성장 기록 목록에서 특정 기록을 선택하면 해당 기록의 상세 내용을 확인할 수 있도록 상세 조회 기능을 구현하였다.

상세 페이지의 URL은 기록 번호를 PathVariable로 전달하는 구조를 사용하였다.

```text
/record/{recordNum}
```

예시는 다음과 같다.

```text
/record/1
/record/2
/record/3
```

각 URL의 마지막 값은 조회할 성장 기록의 고유 번호를 의미한다.

---

## 3. 상세 조회 Controller 구성

선택한 성장 기록 번호를 전달받아 상세 정보를 조회하도록 Controller를 구성하였다.

```java
@GetMapping("/{recordNum}")
public String recordDetail(
        @PathVariable Long recordNum,
        HttpSession session,
        Model model
) {
    Member loginMember = getLoginMember(session);

    if (loginMember == null) {
        return "redirect:/login";
    }

    GrowthRecord record =
            growthRecordService.findRecordDetail(
                    recordNum,
                    loginMember.getMemberNo()
            );

    model.addAttribute("record", record);

    return "record/detail";
}
```

단순히 기록 번호만으로 조회하지 않고 로그인 회원 번호도 함께 전달하였다.

이를 통해 다른 사용자가 URL의 기록 번호를 변경하더라도 본인의 기록이 아닌 데이터에는 접근하지 못하도록 처리하였다.

## 4. 성장 기록 상세 조회 Service 구현

Service에서는 기록 번호와 로그인 회원 번호를 모두 이용해 현재 회원이 작성한 기록인지 확인하도록 구성하였다.

```text
기록 번호로 데이터 조회
→ 기록 존재 여부 확인
→ 기록 작성자와 로그인 회원 비교
→ 일치하면 상세 정보 반환
→ 일치하지 않으면 접근 제한
```

회원별 데이터 보호를 위해 단순한 `findById()`만 사용하는 것보다 기록 소유자까지 함께 확인하는 구조를 적용하였다.

## 6. 성장 기록 내용의 과도한 들여쓰기 수정

상세 페이지에서 성장 기록 내용의 첫 줄이 지나치게 들여쓰기 된 것처럼 출력되는 문제가 발생하였다.

### 원인

JSP에서 `<c:out>` 또는 EL 표현식을 여러 줄로 작성하면서 태그 내부에 포함된 공백과 줄바꿈이 화면에 함께 반영될 가능성이 있었다.

예를 들어 다음과 같이 작성하면 JSP 코드의 줄바꿈과 공백이 내용 앞에 포함될 수 있다.

```jsp
<div class="record-content">
    ${record.content}
</div>
```

또한 CSS에서 `white-space` 속성을 사용하고 있다면 이러한 공백이 그대로 화면에 표시될 수 있다.

### 수정

출력 표현식 앞뒤에 불필요한 줄바꿈이나 공백이 포함되지 않도록 한 줄로 정리하였다.

```jsp
<div class="record-content">${record.content}</div>
```

기록 내용의 줄바꿈을 유지하기 위해 CSS는 다음과 같은 형태로 적용하였다.

```css
.record-content {
    white-space: pre-wrap;
    word-break: break-word;
}
```

### 결과

* 기록 내용 첫 줄의 과도한 들여쓰기 제거
* 사용자가 작성한 줄바꿈 유지
* 긴 문장도 화면 너비에 맞게 자동 줄바꿈

## 7. 성장 기록 수정 기능 설계

성장 기록 상세 페이지에서 기존 내용을 수정할 수 있도록 수정 기능을 구현하였다.

수정 기능의 흐름은 다음과 같이 구성하였다.

```text
상세 페이지에서 수정 버튼 선택
→ 기존 기록 조회
→ 수정 페이지에 기존 값 출력
→ 사용자가 내용 변경
→ 수정 요청 전송
→ DB 데이터 변경
→ 수정된 상세 페이지로 이동
```

수정 페이지는 기록 번호를 URL에 포함하는 구조로 설정하였다.

```text
GET /record/{recordNum}/edit
POST /record/{recordNum}/edit
```

## 8. 수정 페이지 이동 Controller 구현

기존 성장 기록 데이터를 수정 화면에 출력하기 위해 GET 요청을 구성하였다.

```java
@GetMapping("/{recordNum}/edit")
public String editPage(
        @PathVariable Long recordNum,
        HttpSession session,
        Model model
) {
    Member loginMember = getLoginMember(session);

    if (loginMember == null) {
        return "redirect:/login";
    }

    GrowthRecord record =
            growthRecordService.findRecordDetail(
                    recordNum,

            );

    model.addAttribute("record", record);

}
```

수정 페이지에 기존 기록 정보를 전달하여 사용자가 처음부터 다시 입력하지 않아도 되도록 하였다.

## 10. 성장 기록 수정 처리 구현

수정 폼에서 전달한 데이터를 받아 기존 Entity의 값을 변경하도록 POST 요청을 구현하였다.

```java
@PostMapping("/{recordNum}/edit")
public String updateRecord(
        @PathVariable Long recordNum,
        @ModelAttribute GrowthRecordRequest request,
        HttpSession session
) {
    Member loginMember = getLoginMember(session);

    if (loginMember == null) {
        return "redirect:/login";
    }

    growthRecordService.updateRecord(
            recordNum,
            loginMember.getMemberNo(),
            request
    );

    return "redirect:/record/" + recordNum;
}
```

수정 완료 후에는 목록 페이지가 아니라 수정된 기록의 상세 페이지로 이동하도록 처리하였다.

이를 통해 사용자가 변경 결과를 바로 확인할 수 있도록 하였다.

## 11. 수정 Service 로직 구현

Service에서는 수정할 기록이 실제로 존재하는지 확인하고, 해당 기록의 작성자가 현재 로그인한 회원인지 검증하였다.

검증이 완료된 이후에만 Entity의 값을 변경하도록 구성하였다.

```text
기록 조회
→ 로그인 회원의 기록인지 확인
→ 제목 수정
→ 내용 수정
→ 날짜 수정
→ 목표 연결 수정
→ 체감 난이도 수정
→ 트랜잭션 종료 시 변경 내용 반영
```

JPA의 변경 감지를 이용하여 조회한 Entity의 값을 변경하고 별도의 신규 데이터가 생성되지 않도록 하였다.

## 13. 성장 기록 목록 페이지 404 오류 해결

수정 기능 구현 후 성장 기록 목록 페이지로 이동하는 과정에서 다음과 같은 오류가 발생하였다.

```text
There was an unexpected error
type=Not Found
status=404

No static resource record/list.
```

### 원인

브라우저에서는 `/record/list`로 요청하고 있었지만 Controller에는 해당 URL을 처리하는 매핑이 존재하지 않았다.

기존 성장 기록 목록 Controller가 다음과 같이 설정되어 있었다면 실제 목록 URL은 `/record`이다.

```java
@RequestMapping("/record")
```

```java
@GetMapping
public String recordList() {
    return "record/list";
}
```

여기서 `record/list`는 브라우저 요청 주소가 아니라 반환할 JSP View 이름이다.

### 수정

목록 페이지로 이동하는 링크와 리다이렉트 경로를 Controller 매핑에 맞춰 `/record`로 통일하였다.

```jsp
<a href="${pageContext.request.contextPath}/record">
    목록으로
</a>
```

```java
return "redirect:/record";
```

### 결과

* 성장 기록 목록 페이지 정상 이동
* URL과 View 경로의 역할 구분
* 수정 및 삭제 이후 목록 이동 정상화

## 20. 오늘 발생한 트러블슈팅

오늘 개발 과정에서는 다음 문제를 해결하였다.

### 상세 내용 첫 줄 들여쓰기 문제

JSP 출력 태그 주변의 불필요한 공백과 줄바꿈을 제거하고 `white-space: pre-wrap`을 적용하여 사용자가 입력한 형태대로 출력되도록 수정하였다.

JSP View 경로인 `record/list`를 브라우저 URL처럼 사용하여 발생한 문제였다.

실제 Controller 매핑에 맞게 `/record`로 URL을 통일하여 해결하였다.

### 성장 기록 CRUD 미완성 문제

기존에는 등록과 조회만 가능했으나 수정 및 삭제 기능을 추가하여 전체 CRUD를 완성하였다.

## 23. 오늘 배운 점

* CRUD 기능은 등록과 조회뿐 아니라 수정과 삭제까지 모두 구현해야 하나의 기능이 완성된다.
* JSP View 이름과 브라우저가 요청하는 URL은 서로 다른 개념이다.
* `return "record/list"`는 JSP를 찾기 위한 View 이름이며, 반드시 `/record/list` URL을 의미하지는 않는다.
* `<textarea>`의 기존 값은 `value` 속성이 아니라 태그 내부에 작성해야 한다.
* JPA에서는 조회한 Entity의 값을 변경하면 변경 감지를 통해 수정 내용을 반영할 수 있다.
* 수정과 삭제 기능에서도 로그인 회원과 데이터 소유자가 같은지 반드시 확인해야 한다.
* 삭제 기능은 사용자의 실수를 줄이기 위해 확인 절차를 추가하는 것이 좋다.
* 사용자 입력의 줄바꿈을 유지하려면 `white-space: pre-wrap`을 활용할 수 있다.
* 수정 완료 후 상세 페이지, 삭제 완료 후 목록 페이지로 이동하면 사용자 흐름이 자연스럽다.
